# 部署实施方案

## 环境准备

Linux 服务器、Docker/Compose、HTTPS 域名、PostgreSQL、Redis、模型 API Key。复制 `.env.example` 为 `.env`，替换数据库密码、JWT 密钥、管理员账号和模型配置。

```bash
docker compose build
docker compose up -d
docker compose ps
curl -f http://127.0.0.1:8080/health
```

生产必须配置：`APP_ENV=production`、`AUTH_ENABLED=true`、强 `AUTH_JWT_SECRET`、`AUTH_COOKIE_SECURE=true`、真实管理员密码和真实模型。不得使用 `change_me`。

## 实施步骤

1. 备份旧版本；确认回滚镜像和数据库备份。
2. 部署数据库/Redis并完成健康检查。
3. 执行 `alembic upgrade head`；Docker 入口会在 API 启动前自动执行。
4. 部署 API，执行冒烟、权限和模型测试。
5. 灰度导入知识，观察拒答率、延迟和成本。
6. 切换流量，保留旧版本至少一个回滚周期。

回滚时先切回旧镜像；数据库仅使用经过验证的向下迁移或备份恢复，禁止直接删除卷。
# 2026-06-30 一键部署补充

开发或演示环境在项目根目录执行 `docker compose up -d --build`。系统自动启动 API、PostgreSQL/pgvector、Redis、Mock OMS、Prometheus 和 Grafana，并自动执行 Alembic 升级。验收地址分别为 `8080`、`8090`、`9091` 和 `3001` 端口。

生产部署先复制 `.env.production.example` 为服务器本地 `.env`，替换所有 `REPLACE_` 值并设置 `COMPOSE_DATABASE_URL`。密钥不得进入 Git。GitHub Actions 的 `Deploy Demo` 工作流需要配置 `DEMO_HOST`、`DEMO_USER`、`DEMO_SSH_KEY`、`DEMO_PATH` 四项 Environment Secret，执行后会上传发布文件、重建服务并调用 `/health` 验收。
