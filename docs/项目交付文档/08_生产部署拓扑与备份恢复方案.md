# 生产部署拓扑与备份恢复方案

```text
用户 → WAF/负载均衡/HTTPS → API 实例×2+
                              ├→ PostgreSQL 主库/备库
                              ├→ Redis 主从/哨兵
                              ├→ 向量库（规划）
                              ├→ OMS/ERP/WMS
                              └→ 多模型网关
监控：指标、日志、链路、告警 → 独立观测平台
```

API 无状态横向扩展；Cookie JWT 不依赖本地会话。数据库、缓存和日志不得暴露公网。跨可用区部署，入口启用限流和健康摘除。

## 备份策略

- PostgreSQL：每日全量、持续 WAL/PITR，保留 30 天；备份加密并异地保存。
- 配置：版本化保存脱敏配置，密钥由密钥管理系统托管。
- 知识文件：对象存储版本控制与生命周期策略。
- 审计：至少保留 180 天或满足企业合规要求。

建议 RPO≤15 分钟、RTO≤2 小时。每季度恢复演练：新环境恢复数据库→校验行数和租户隔离→启动应用→执行核心用例→形成演练报告。Redis 缓存可重建，不作为唯一数据源。
# 2026-06-30 可观测性拓扑补充

Prometheus 每 10 秒抓取 `api:8080/metrics`，Grafana 已自动预置 Prometheus 数据源和“电商智能客服运行概览”仪表盘，展示 QPS、P95 延迟和 HTTP 错误率。API 日志采用单行 JSON 并包含 `request_id`、方法、路径、状态码和耗时，生产环境应由容器日志驱动采集至 Loki、ELK 或云日志服务。

PostgreSQL 和 Redis 使用命名卷保存数据；上线前必须为 PostgreSQL 配置定时全量备份、WAL/PITR 和异地副本，并至少完成一次从空环境恢复演练。`docker compose down` 不删除数据卷，禁止在未确认备份的情况下执行 `docker compose down -v`。
